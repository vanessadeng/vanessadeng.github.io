MODULE 1

LECTURE NOTE-1: Introduction to Software Engineering

The term software engineering is composed of two words, software and engineering.

  • Software is more than just a program code. A program is an executable code, which serves some computational purpose. Software includes executable programming code, associated libraries, and documentations. When made for a specific requirement, it is called a software product.
  • Engineering is about developing products using well-defined, scientific principles and methods.

We define software engineering as an engineering branch associated with the development of software products using well-defined scientific principles, methods, and procedures. The outcome of software engineering is an efficient and reliable software product.

IEEE defines software engineering as:
“The application of a systematic, disciplined, quantifiable approach to the development, operation, and maintenance of software.”

We view it as a systematic collection of past experience, arranged in the form of methodologies and guidelines. Using software engineering principles is necessary to develop a large, high-quality software cost-effectively.

Software engineering principles help reduce the complexity of programming. They employ two important techniques: abstraction and decomposition.

  • Abstraction means simplifying a problem by omitting irrelevant details.
  • Decomposition involves dividing a complex problem into smaller parts, which can be solved independently and then integrated into a full solution.

NEED OF SOFTWARE ENGINEERING

The need for software engineering arises due to the higher rate of change in user requirements and the environment on which the software operates.

  • Large software: As the size of software becomes large, engineering must step in to apply a scientific process.
  • Scalability: Without scientific and engineering concepts, it would be easier to re-create new software than to scale an existing one.
  • Cost: Although the cost of hardware has decreased due to mass production, the cost of software remains high if proper processes are not followed.
  • Dynamic Nature: The continuously changing nature of software depends heavily on the environment in which the user operates.
  • Quality Management: A better software development process leads to a higher quality product.

CHARACTERISTICS OF GOOD SOFTWARE

A software product can be judged by what it offers and how well it can be used. This software must satisfy the following criteria:

  • Operational: How well the software works in operations. It includes factors like budget, usability, efficiency, correctness, functionality, dependability, security, and safety.
  • Transitional: Important when the software is moved from one platform to another. It involves portability, interoperability, reusability, and adaptability.
  • Maintenance: How well the software can maintain itself in an ever-changing environment. This includes modularity, maintainability, flexibility, and scalability.

Software engineering is a branch of computer science that uses well-defined engineering concepts to produce efficient, durable, scalable, budget-friendly, and on-time software products.

LECTURE NOTE-2: Software Development Life Cycle (SDLC)

The Software Development Life Cycle (SDLC) is a structured sequence of stages in software engineering to develop the intended software product efficiently. SDLC activities include several steps:

SDLC Framework:

  • Communication: The user initiates the request for a desired software product, contacts the service provider, and tries to negotiate the terms.
  • Requirement Gathering: The software development team works to extract as much information as possible from various stakeholders. This involves studying existing systems, conducting interviews, referring to databases, and collecting questionnaire responses.
  • Feasibility Study: The team evaluates whether the project is feasible financially, practically, and technologically. If feasible, they proceed; otherwise, the project is dropped.
  • System Analysis: Developers create a roadmap for the project, deciding on the best software model and planning resources and schedules based on the project’s scope.
  • Software Design: Based on gathered requirements and system analysis, the software product is designed. This stage results in logical and physical designs through data dictionaries, logical diagrams, and data-flow diagrams.
  • Coding: Known as the programming phase, where developers write code in the chosen programming language and develop executable programs.
  • Testing: Extensive testing is conducted to ensure the software meets all specifications and requirements. This includes module testing, program testing, and in-house testing.
  • Integration: Integration of software with existing systems, libraries, and databases.
  • Implementation: Installing the software on user machines, including necessary configurations.
  • Operation and Maintenance: Ensuring the software operates efficiently with fewer errors. This may involve training users and updating the software to accommodate changes in its environment.
  • Disposition: Phasing out a system when it becomes obsolete or requires substantial upgrades, including data archiving and system shutdown.

The Need for a Software Life Cycle Model:

To ensure systematic and disciplined development, a suitable life cycle model must be identified and followed. Without a specific model, development can lead to chaos and project failure. A life cycle model specifies entry and exit criteria for each phase, aiding in the systematic progression through the SDLC.

LECTURE NOTE-3: Prototyping Model and Spiral Model

Prototyping Model:

A prototype is an early sample, model, or release of a product built to test a concept or process. It is a term used in various contexts, including semantics, design, electronics, and software programming. A prototype is generally used to evaluate a new design to enhance precision by system analysts and users. Prototyping serves to provide specifications for a real, working system rather than a theoretical one. Steps of the prototyping model include:

  1. Basic Requirement Identification: Focus on essential requirements, particularly user interfaces.
  2. Developing the Initial Prototype: The initial prototype is created incorporating the basic requirements.
  3. Review of the Prototype: This involves user and stakeholder feedback, which is crucial for further development.
  4. Revise and Enhance the Prototype: Based on feedback, the prototype is refined and enhanced repeatedly until it meets all specified requirements.

Spiral Model:

The Spiral Model combines elements of both design and prototyping in stages, in an effort to combine advantages of top-down and bottom-up concepts. It is especially useful for large projects involving complex systems. The model allows for incremental releases of the product, or incremental refinement through each iteration around the spiral.

  • Planning: Identifying resources, timelines, and other project-related information.
  • Risk Analysis: Analyzing and addressing potential risks at each step of the project.
  • Engineering: Building one or more forms of the product for further testing and validation.
  • Evaluation: Evaluating the results of testing of the prototypes at the end of each iteration.

These lecture notes provide a solid foundation in understanding the various models and approaches within the field of software engineering. Each model or approach has its merits and can be chosen based on specific project requirements and contexts.

LECTURE NOTE-4: Requirement Analysis & Specification

Requirement Analysis is critical for the success of a software project. It involves defining user expectations for a new or modified product. This step ensures that the final software product meets all user and system requirements.

Requirement Engineering Process:

This process involves several crucial steps:

  • Feasibility Study: Determines if the project is financially and technically feasible.
  • Requirement Gathering: Direct interaction with clients to gather all detailed requirements.
  • Software Requirement Specification (SRS): Documenting the gathered requirements clearly and precisely.
  • Software Requirement Validation: Ensuring the documented requirements are feasible and within the scope of the project.

Goals of Requirement Engineering:

To produce a comprehensive document that:

  • Is clear and understandable to all stakeholders.
  • Provides a base for creating the software design.
  • Acts as a reference tool for future maintenance and enhancements.

LECTURE NOTE-5: Software Requirement Characteristics

A well-defined SRS (Software Requirement Specification) must be:

  • Clear
  • Correct
  • Consistent
  • Coherent
  • Comprehensible
  • Modifiable
  • Verifiable
  • Prioritized
  • Unambiguous
  • Traceable
  • Credible source

Types of Requirements:

  • Functional Requirements: Define the fundamental actions that the system must perform.
  • Non-Functional Requirements: Include all the system’s operational capabilities, like security and usability.

User Interface (UI) Requirements:

A critical aspect that significantly affects user satisfaction. It should be:

  • Easy to navigate.
  • Responsive.
  • Consistent across different screens.
  • Capable of handling errors effectively.

LECTURE NOTE-6: Software Design

Software Design is the process of envisioning and defining software solutions to one or more sets of problems. This stage helps transform the requirement specifications into a blueprint for constructing the software.

Software Design Levels:

  • Architectural Design: The broadest view of the system. It defines the structure and high-level components.
  • High-Level Design: Refines the architectural design into a more detailed breakdown.
  • Detailed Design: Drills down into the high-level design to define all technical details and processing logic.

Modularization:

Crucial for managing complexity. Modules are self-contained units of software that handle specific tasks.

LECTURE NOTE-7: Cohesion & Coupling

These are two fundamental concepts used to evaluate the quality of a software design.

  • Cohesion: Refers to the degree to which the elements of a module belong together. High cohesion within modules is desired.
  • Coupling: Refers to the degree to which modules depend on each other. Systems with low coupling are preferred because they are easier to manage and modify.

LECTURE NOTE-8: Software Analysis & Design Tools

Data Flow Diagrams (DFD):

Used to represent the flow of data through a system and the work or processing performed by each module.

Structure Charts:

Show the hierarchy and structure of modules in the software. Useful for understanding the overall system architecture.

Structured English and Pseudo Code:

Help bridge the gap between the technical and business views of a system by describing the logic in plain language.

LECTURE NOTE-9: Decision Tables

Decision Tables are a precise yet straightforward way to model complicated logic. They provide a clear way of stating complex decision-making processes that involve numerous rules and conditions.

LECTURE NOTE-10: Entity-Relationship (ER) Model and Data Dictionary

ER Model:

Visualizes entities, their relationships, and key attributes. It’s fundamental for database design.

Data Dictionary:

Acts as a catalog of all data elements, containing details about data structure, format, and relationships. This tool is essential for consistency across the system.
